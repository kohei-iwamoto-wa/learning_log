### EKS Overview

kubernetes は、コンテナオーケストレーションプラットフォームです。

#### Kubernetes が必要な理由

コンテナが数個レベルであれば、手動管理できますが、数百個以上になると管理が厳しくなります。

※ 実際には数十個単位でも管理が厳しくなると思われます。

* Kubernetes の主要機能
  * 自己修復
    * コンテナが停止すると自動的に新しいコンテナを立ち上げる
  * スケーリング
    * アクセス数に合わせてオートスケーリングを行う
  * ロードバランシング
    * 複数のコンテナにトラフィックを振り分けます。
  * 宣言的設定
    * desired に 3 を設定していると、システムが desired の数に合わせてコンテナを起動する。

### コンテナを複数台起動したいなら docker compose でよくない？

コンテナを複数台起動したいなら docker compose でよくないと思っていました。

ただ、docker compose は、コンテナの起動やコンテナ間のネットワークを簡単にたちあげることはむいています。

docker compose にできないことは、オートスケーリングです。トラフィックが増えた場合でもコンテナの自動的に増やすことはできないので、可用性が求められる場合は、利用は控えたら良いです。

また、docker compose は単一ホストでのみ対応します。なので複数ノードをグループ化してコンテナを起動したいケースは対応できません。

docker compose を使えば、コンテナの管理楽だなと思っていましたが、可用性の顧慮が必要ある環境ではつかえないでしょう。

docker compose が開発環境では使われてるが、本番環境では使いづらい理由は可用性面が大きいです。

ただ、少数のコンテナのみの環境やダウンタイムが起動できる場合は利用してもいいかもしれません。
  
※ docker compose はコンテナオーケストレーターではなくコンテナ管理ツールです。

### CPU アーキテクチャとイメージのビルド

異なるCPU アーキテクチャでビルドしたイメージは Run できない。

例）x86_64 でビルドした arm64 では利用できない。 

#### 理由　

CPU アーキテクチャが異なるとマシン語が異なるため

#### 対策

* 社内の開発 PC や サーバ の CPU アーキテクチャを一致させておく。
* マルチプラットフォームビルド
  * docker buildx build --platform linux/amd64,linux/arm64 -t username/repo:latest --push .

### Arm

#### Arm の特徴

* 消費電力が従来のプロセッサーより少ない
* RISK のアーキテクチャなので高速処理できる
* AWS で Armベースのプロセッサーである Graviton が開発されクラウドでの採用が加速している
  * Arm は、スマートフォン、IoT 機器であった圧倒的なシェアを占めている。

#### デメリット

* Intel 製のプロセッサーとの互換性不足
  * 高いパフォーマンスがあるが、Intel 製との互換性がない場合があったり、Python のライブラリでも Arm に互換性がない場合がある。
* Intel から乗り換える際は、動作テストは念入りに行う必要がある。
  
### 活用

Kubernetes ではノードごとに CPU のアーキテクチャを指定できる。

A ノードでは、Intel, B ノードでは、Arm など分けることができる。

高速なCPU を利用したい場合は, Arm搭載の B ノードに Pod に配置し、その他は A ノードに配置するなどできます。

Taint / Toleration の設定で対応可能です。
