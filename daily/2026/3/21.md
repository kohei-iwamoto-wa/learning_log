### requests と limits 

Pod のリソース管理（メモリ・CPU）に関連する要素の requests と limits について

Pod に requests を設定すると、コンテナは requests を設定された値よりも多くのリソースを設定することができます。

例えば、256 MB と コンテナは 512 MB のように設定された requests よりも多くのリソースを利用します。

Limit を設定すると、コンテナは、limit で設定された値よりも少ないリソースしか利用できないようになります。

ユースケースは、requests では、コンテナ稼働のための最低限のリソースを設定することで、ベースを保証します。

limits は、想定以上コンテナがリソースを使いすぎることのないように抑制することで、他のコンテナの稼働に必要なリソースを利用しないことを保証するために利用します。

#### ポイント

* requests を超えそうな場合
  * スケジューラが Pod をどのノードに配置するか決定する基準になります。
* limits
  * CPUの場合： スロットリングが発生し動作が遅くなります。
  * メモリの場合： OOMKiller が発動して プロセス（コンテナ）が停止します。

#### メモリの limits を低く設定する

```
apiVersion: v1
kind: Pod
metadata:
  name: frontend
spec:
  containers:
  - name: app
    image: nginx:latest
    resources:
      requests:
        memory: "2Mi"
        cpu: "250m"
      limits:
        memory: "5Mi"
        cpu: "500m"
```

これをを適用すると、コンテナが limits 以上のリソースを利用しようとしたので、OOMKiller が発動する。

```

default              frontend                                     0/1     OOMKilled   4 (63s ago)   114s
```

#### manifest 再適用時のエラー

稼働中 Pod の設定値を kubectl apply -f で変更しようとした時、以下のようなエラーが出力された。

```
xxx@yyy:~/projects/learning_log/daily/2026/3$ kubectl apply -f request_limit.yaml 
The Pod "frontend" is invalid: spec: Forbidden: pod updates may not change fields other than `spec.containers[*].image`,`spec.initContainers[*].image`,`spec.activeDeadlineSeconds`,`spec.tolerations` (only additions to existing tolerations),`spec.terminationGracePeriodSeconds` (allow it to be set to 1 if it was previously negative)
```

稼働中のPodの設定変更できる設定値には制限があります。

* Pod 稼働中に変更できる項目
  * spec.containers[*].image
  * spec.initContainers[*].image
  * spec.activeDeadlineSeconds
  * spec.tolerations
  * spec.terminationGracePeriodSeconds

これ以外の項目は、pod を一旦削除してから変更する必要がある。
