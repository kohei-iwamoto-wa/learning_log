### 学んだこと

`aws-auth ConfigMap` は、IAM と Kubernetes の RBAC をバインディングする手段です。

クライアントがクラスターへ接続するための認証方式のこといいます。

kubectl でクラスターに接続する際、 `~/.kube/config` の以下の情報を利用します。

`admin` が `my-cluster.region-code.eksctl.io` へのアクセス権限があることを意味しています。

通常、kubectl で クラスタに アクセスできるユーザは、クラスタの作成者のみです。

```
contexts:
- context:
    cluster: my-cluster.region-code.eksctl.io
    user: admin@my-cluster.region-code.eksctl.io
  name: admin@my-cluster.region-code.eksctl.io
current-context: admin@my-cluster.region-code.eksctl.io
```

aws-auth ConfigMap を利用することで、別ユーザーに権限を与えることができる。

ただ、`aws-auth ConfigMap`  を使った認証方式は、非推奨になった。

非推奨の理由は、`aws-auth ConfigMap` の設定ミスをすることでadmin ユーザーもクラスタにアクセスできなくなる恐れがあるため。

YAML の構文エラー: aws-auth は Kubernetes の ConfigMap というリソースです。編集時にスペース 1 つのインデントを間違えたり、不要な文字が混じったりした状態で kubectl apply してしまうと、ConfigMap 自体が壊れて読み込めなくなります。

### 認証の流れ

1. IAM ユーザー/ロールの作成: AWS 側で新しい担当者用の IAM を用意する。
2. ConfigMap への追記: aws-auth にその IAM ARN と、対応させる Kubernetes 内の「グループ名」を書き込む。
3. RBAC の設定: ClusterRoleBinding などを作成し、その「グループ名」に対して具体的な権限（Pod の閲覧など）を紐付ける。

※　グループ名は RBAC roleBInding の group

### 今日の気づき

`aws-auth ConfigMap`  の設定変更は高リスクがあり、管理者すらアクセスできない状態になる。

そのため、`aws-auth ConfigMap` は非推奨となっている。

その代わりに Access Entry が推奨されている。

Access Entry については別途学習が必要
